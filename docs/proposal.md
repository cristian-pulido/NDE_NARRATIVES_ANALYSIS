# Structured Coding of NDE Narratives with Human and LLM Annotation

## Objective

This repository supports the analysis of narrative reports of Near-Death Experiences (NDEs) collected in a survey dataset. The goal is to extract structured information from three narrative sections so those annotations can be compared with questionnaire responses, VADER-derived tone signals, and outputs generated by multiple large language model experiments.

## Narrative Units

Each participant provides three sections:

1. `context`: circumstances surrounding the event
2. `experience`: phenomenological description of the NDE
3. `aftereffects`: perceived consequences following the experience

The current study design assumes complete narratives across all three sections.

## Measures Linked to the Narratives

- `NDE-C`: phenomenological characteristics of the NDE
- `LCI-R`: long-term changes after the experience

## Structured Outputs

The coding framework extracts:

- Tone for each section: `positive`, `negative`, or `mixed`
- Ten NDE-C elements from the `experience` section.
- Ten LCI-R elements from the `aftereffects` section.

All extracted elements are binary: `yes` or `no`.

## Why the Repository Uses Three Separate Prompts

The source dataset contains three distinct narrative sections that correspond to different stages of the experience. The repository therefore keeps prompt generation and evaluation section-specific instead of concatenating the narratives into a single block.

This makes it possible to analyze trajectories such as:

- negative context -> positive experience
- positive experience -> positive aftereffects
- mixed trajectories

## Questionnaire Binarization Rules

### NDE-C

- `Medium`, `Intensively`, `Extremely` -> `yes`
- `Not at all - absence`, `Slightly` -> `no`

### LCI-R

- Any explicit change (`Increased`, `Strongly increased`, `Decreased`, `Strongly diminished`) -> `yes`
- `Not changed` -> `no`
- `Missing` or blank -> excluded (`NA`)

## Multi-Annotator Design

The repository now assumes that more than one human annotator may code the same sampled dataset.

The evaluation workflow therefore separates three layers:

- raw annotator workbooks
- a consolidated majority-vote human reference
- downstream comparisons against questionnaire, VADER, and LLM experiments

This preserves traceability while still producing a single reference set for evaluation.

## Experiment Design

The repository also assumes that LLM outputs may come from multiple experiments.

An experiment can be identified by:

- `experiment_id`
- optional `prompt_variant`
- optional `run_id`
- optional `model_variant`

This allows prompt variants and repeated model runs to remain distinguishable in outputs and reports.

## Evaluation

The repository calculates:

- Accuracy
- Cohen's Kappa
- Macro F1

The intended comparisons are now:

- majority human reference vs. questionnaire-derived binary labels
- majority human reference vs. VADER-derived tone labels
- majority human reference vs. each accepted LLM experiment
- pairwise human annotator agreement summaries

LLM-vs-LLM comparisons are intentionally out of scope in the current phase.
